Command Line Cheat Sheet

Below is a brief list of fundamental Command Line commands and a brief description of what they do and any notes on correct syntax.

Navigation

The commands here are based around basic navigation through the file system using the command line.

Tag Name Description
ls List This command tells the computer to list all of the files and directories in the current directory.
pwd Print Working Directory This command tells the computer to print the current directory. This enables you to locate where you are in the system file tree.
cd Change Directory This command tells the computer to change into a different directory. The syntax to change directory would be cd and then the name of the directory you wish to change into.
.. Move up a level This is not a command in the same way as the others but is a shorthand that tells the computer to change into the directory above the current one. It is used as an addition to another command so for example the syntax to change into the above directory would be cd and then .. instead of having to write the longhand of the directory name each time you want to move up a level.
mkdir Make Directory This command tells the computer to create a new directory. The command should be followed by a space and then the name of the directory to be created. The command mkdir folder, would make a new directory called "folder" in the current directory within the file system.
touch Make File This command tells the computer to create a new file. The command should be followed by a space and then the name of the file, including suffix, to be created. The command touch file.txt, would make a new text file called "file" in the current directory within the file system.

Git

The commands here are general commands for working within the Git workflow.

Tag Name Description
git init Initialize This command creates a new Git repository.
git status Check Status This command inspects the contents of the working directory and staging area.
git add Add This command allows adding of files from the working directory to the staging area.
An example of the syntax would be;
git add filename
When noting the filename this needs to also include the suffix such as filename.txt

Multiple files can be added to the staging area this way, they can be chained in one command, for example;
git add filename_01 filename_02

A shortcut to add all files in the working directory is;
git add .
git diff Difference This command allows for a comparison between files in the working directory and those in the staging area. Differences are highlighted in green.

An example of this command in action is explained below.
In the staging area there is an existing file called file-01.txt, in order to compare that with the version contained in the working directory, we would write git diff file-01.txt.
This would return a comparison between the 2 files, highlighting differences.
git commit Commit This command permanently stores file changes from the staging area to the repository.
The syntax includes another small section of code and a very brief description contained within quotation marks. An example is shown below;
git commit -m "Commit file to repository"

There are standard conventions for commit messages, they must be in quotations, they must be written in the present tense and they should be brief (50 characters or less) when using -m.
git log View Log This command pulls up all earlier versions of a project to view.
The contained information of a log will show the unqiue commit code (a 40 character SHA), the commit author, the date and time of the commit and the commit message.
git log --oneline View Log in one line format This command does the same as git log but is a condensed format that shows the first 7 character of the SHA and comment of the commit rather than the full body of information found in a normal log command.
git log -s "keyword" View Commits with given Keyword This command displays a list of commits where the number of occurrences of the keyword changes within at least one file via addition, deletion, or modification.
git checkout HEAD filename Discard working directory changes This command resets the file in the working directory to the file that was most recently commited. This is used primarily in case changes were made to the file in the working directory that are no longer required or relevant.
git reset HEAD filename Unstage files This command allows for the removal of files that have been staged but not commited. This is useful if a file has been staged and further changes are required before being commited or if the wrong file has been staged.
git reset commit_SHA Reset to a previous commit This command resets the current HEAD (most recent commit) to a previous commit in the history of the project. This allows for rolling the project back to a previous point if needed.
The commit_SHA section of the syntax is a placeholder. When using this command instead of using commit_SHA you put in the first 7 characters of the commit's SHA number you wish to roll back to.
git commit --amend Amend a commit This command allows you to correct mistakes and edit commits instead of creating new ones.

It’s important to note that although it seems like --amend is simply updating the commit, what Git actually does is replace the whole previous commit. For this reason, when you execute the command git commit --amend, your terminal editor asks you to update your commit message. However, if you want to keep the same commit message, you can simply add the flag --no-edit
This results in the syntax;
git commit --amend --no-edit
git push Push This command uploads local repository content to a remote repository such as GitHub.
The syntax for pushing is;
git push <remote repository name> <branch name>

Git Branches

The commands here are additional commands for working within the Git workflow.

Tag Name Description
git branch Branch check This command tells you the name of the branch you are currently on.
git branch new_branch Create new branch This command allows you to create a new branch where the new_branch section of the command is the name of the new branch you are creating.

Branch names cannot contain white spaces. For example new-branch or new_branch would be valid names.
git checkout branch_name Change branch This command allows you to change branches.

When working on another branch you can make commits to the branch without the master branch being affected.
git merge branch_name Merge branch This command allows you to merge branches.

When a branch is publish ready you can merge it to the master by navigating into the master branch and then using the command targetting the branch to be merged.
Merge conflicts When working on a new branch from the master branch there will be a time when the code is ready to be integrated into the master branch. We do this with the git merge branch_name command, however what if there have been additional commits to shared files on the master branch while we were working on the new branch?
When merging there will be a conflict between files. One file will be the updated file and one will not.
Git doesn't automatically know what to do with these conflicts so instead it will fail the merge, giving an error message;

CONFLICT (content): Merge conflict in FILENAME.txt
Automatic merge failed; fix conflicts and then commit the result.

It will also mark in the conflicting file(s) the areas of conflict.
For example, lets say we have a file named form.js in our master branch. We also have this same file in a secondary branch named new_form that we've made to work on new features without affecting the site.

Our new_form branch is complete and ready to be merged but while we've been working on that, there have been a few minor changes to form.js on the master branch.
When we go to merge the branch we will get an error message saying the conflict is in form.js and that we need to fix the conflict. The merge will not happen.
In the conflicting file there will now be a marker indicating the conflict(s). It will look like the below, where HEAD refers to the master branch;

<<<<<<< HEAD
master version of line
=======
new_form version of line
>>>>>>> new_form

Once the conflicts are resolved, the changes to that specific file can be staged and then commited.
The branches now have no conflicts and can be merged.
git branch -d branch_name Delete branch This command allows you to delete branches.

When a branch has been merged and it no longers needs to be worked on it can be deleted.

Git Teamwork

The commands here are additional commands for working within the Git workflow.

Tag Name Description
git clone remote_location clone_name Clone repository This command allows you to make a local clone of a repository.

Within this command romte_location tells Git where to go to find the remote. This can be a web address or a filepath such as /Users/username/Documents/folder_location

clone_name is the name that you give to the directory into which Git will clone the repository.

An important point to note here is that if you commit any changes to the project in your local storage folder then the rest of the team do not have access to this and therefore do not know about the changes.
git remote -v Origin and viewing remotes When you clone a repo Git automatically gives the remote address the name origin. This is to allow for it to be referred to easily.

Running the command git remote -v within the clone folder will allow you to view the projects remotes and it will give a read out of the fetch and push locations.
They will look something similar to the below;

origin /Users/username/Documents/folder_location (fetch)
origin /Users/username/Documents/folder_location (push)

The above are just example locations but they could also be web addresses as well as filepath locations.
git fetch Fetching remote data and the general workflow When working on new features in different branches the master branch will potentially be updated along the way.
We need to keep our local clone copy of the master branch up to date within our local clone to ensure that whatever we are working on is working with the current version of the master branch. This, in turn, will ensure it works when we merge it to the master branch upon completion.

We do this by using git fetch to fetch the updated data.
An important point to note here however is that when we use git fetch it will fetch the latest commits of the origin/master branch, our local master branch has not been updated with this data yet.
We can use the command git merge origin/master to update this. Once this has been carried out our local master branch will be up to date with the origin.

Now the local master branch is up to date we follow the below workflow;

1 - Fetch and merge changes from the remote.
2 - Create a branch to work on a new project feature.
3 - Develop the feature on your branch and commit your work.
4 - Fetch and merge from the remote again (in case new commits were made during work on the new feature).
5 - Push the new branch up to the remote for review.

Push is explained later but these steps illustrate, generally, how to collaboratively work on a larger project.
git push origin branch_name Pushing new branches for review When a feature has been completed, the local copy of the branch will need to be pushed to the remote (origin) so it can be reviewed by others and then potentially merged into the master if deemed acceptable.
git rebase Rebasing branches Rebasing is an operation possible in Git repo's when the base of a branch needs to be relocated to a new commit point on the masters commit history for some reason. This could be because a bug was found and then fixed or some other changes were made and then merged to the master branch.

Rebasing a branch rewrites its history. It avoids needing to merge back into the master branch and then branching out again into another new feature branch. It keeps a clean history rather than a complicated back and forth.

To rebase a branch, navigate to the branch you want to rebase and then use the command;

git rebase main

This will rebase the branch and rewrite its history and incoporate any changes that the main branch may have undergone into your local main branch. This can be easily checked by inspecting the files and manually noticing any changes to the code.
You can view the rebased branch by logging the commit history of the branch using the command;

git log --graph --decorate --oneline --all

This will display tbe rebased branch ahead of the main branch with all its history relocated to be ahead of the main branch.
